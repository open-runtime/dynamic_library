# Dynamic Library Core - Examples

This document provides practical, copy-paste-ready examples for integrating and using the `dynamic_library` package. 

### 1. Basic Usage

This example demonstrates how to use the core utility methods to resolve library naming, load a dynamic library into memory, and safely wrap it using the `DLib` class for path tracking.

```dart
import 'dart:ffi';
import 'package:dynamic_library/dynamic_library.dart';

void main() {
  // 1. Determine the correct OS extension (e.g., '.dll', '.so', '.dylib')
  String ext = systemLibExtension();
  print('Running on platform expecting $ext files.');

  // 2. Resolve the OS-specific library name
  // Windows: 'foo.dll', Unix: 'libfoo.so', MacOS: 'libfoo.dylib'
  String libName = fullLibraryName('foo');
  print('Standard library file name: $libName');
  print('Standard library prefix: ${libraryPrefix()}');

  // Check if we are running in a Dart context vs a compiled Flutter app
  if (isDart()) {
    print('Running from dart executable.');
  }

  try {
    // 3. Load the library into memory
    // Note: Do not include 'lib' or '.so'/'.dll' in the libraryName parameter.
    DynamicLibrary dylib = loadDynamicLibrary(libraryName: 'foo');

    // 4. Wrap the loaded library in the DLib class
    // DLib securely stores the fully resolved file path alongside the DynamicLibrary reference.
    String resolvedPath = fullLibraryPath('foo');
    DLib myLibrary = DLib(resolvedPath, dylib);

    print('Successfully loaded!');
    print('Library Name: ${myLibrary.fileName}');
    print('Library Path: ${myLibrary.path}');
    print('Library Extension: ${myLibrary.extension}');
  } on LoadDynamicLibraryException catch (e) {
    print('Failed to load library "foo".');
    print(e.cause);
  }
}
```

### 2. Common Workflows (Flutter Rust Bridge)

A typical integration pattern is leveraging `initializeFrbLibrary` for `flutter_rust_bridge` projects. It abstracts out cross-platform resolution logic, including standardizing macOS path idiosyncrasies.

```dart
import 'dart:io';
import 'package:dynamic_library/dynamic_library.dart';

// Import the generated ExternalLibrary type from flutter_rust_bridge
import 'package:flutter_rust_bridge/flutter_rust_bridge_for_generated_io.dart' show ExternalLibrary;

// In a real application, this is generated by flutter_rust_bridge.
// We provide a mock here to ensure the code compiles.
class RustLib {
  static Future<void> init({required ExternalLibrary externalLibrary}) async {
    print('Rust Library Initialized!');
  }
}

Future<void> main() async {
  // Example A: Local Dart Development (`dart run` or `dart test`)
  // Uses assumeLocalResolution (defaults to true) to look for libraries relative to the current directory.
  try {
    await initializeFrbLibrary(
      packageName: 'runtime_native_audio',
      rustLibInit: RustLib.init,
      assumeLocalResolution: true,
    );
    print('Initialized local resolution successfully.');
  } on ArgumentError catch (e) {
    print('Local resolution failed (did you run the build script?): $e');
  }

  // Example B: System/App Bundle Resolution (Production)
  // Disable local resolution to look inside standard application framework folders.
  try {
    await initializeFrbLibrary(
      packageName: 'runtime_native_audio',
      rustLibInit: RustLib.init,
      assumeLocalResolution: false, 
    );
    print('Initialized system resolution successfully.');
  } catch (e) {
    print('System resolution failed (expected if not running from a compiled bundle): $e');
  }
  
  // Example C: Direct FRB Naming & Path resolution without initialization
  print('FRB Dynamic Library Name: ${frbDynamicLibraryName('runtime_native_audio')}');
  print('FRB Full Library Path: ${frbFullLibraryPath('runtime_native_audio')}');
}
```

### 3. Advanced Initialization Workflows (Bytes & Paths)

When loading dynamic libraries from assets or custom paths, `initializeFrbLibrary` provides advanced parameters to handle byte buffers or explicit paths.

```dart
import 'dart:io';
import 'dart:typed_data';
import 'package:dynamic_library/dynamic_library.dart';
import 'package:flutter_rust_bridge/flutter_rust_bridge_for_generated_io.dart' show ExternalLibrary;

class RustLib {
  static Future<void> init({required ExternalLibrary externalLibrary}) async {}
}

Future<void> main() async {
  // Example A: Explicit Path Resolution
  // Bypasses the standard resolution and uses an exact path to a file.
  try {
    await initializeFrbLibrary(
      packageName: 'my_package',
      rustLibInit: RustLib.init,
      libPath: '/usr/local/lib/libmy_package.so',
      assumeLocalResolution: false, // Must be false when providing libPath
    );
  } catch (e) {
    print('Explicit path resolution failed: $e');
  }

  // Example B: Byte-based Resolution
  // Useful when loading a library from an asset bundle (e.g. Flutter root bundle).
  // The bytes are temporarily cached to a file so `ExternalLibrary.open` can consume it.
  try {
    ByteData assetBytes = ByteData(0); // Mocked data: await rootBundle.load('assets/libmy_package.so');
    Directory cacheDir = Directory.systemTemp;

    await initializeFrbLibrary(
      packageName: 'my_package',
      rustLibInit: RustLib.init,
      libBytes: assetBytes,
      libBytesCacheDir: cacheDir,
      assumeLocalResolution: false, // Must be false when providing libBytes
    );
  } catch (e) {
    print('Byte-based resolution failed: $e');
  }
}
```

### 4. Error Handling & Dependencies

Dynamic libraries often fail to load not because the `.so` or `.dll` file is missing, but because a downstream C/C++ dependency is missing. The `LoadDynamicLibraryException` automatically detects this and provides rich diagnostics.

```dart
import 'dart:ffi';
import 'package:dynamic_library/dynamic_library.dart';

void main() {
  try {
    // Attempting to load a library that does not exist in the specified searchPath
    DynamicLibrary lib = loadDynamicLibrary(
      libraryName: 'missing_lib', 
      searchPath: '/opt/local/lib',
    );
  } on LoadDynamicLibraryException catch (e) {
    // The LoadDynamicLibraryException captures robust diagnostic data.
    // If the file EXISTS but failed to load, the exception automatically runs
    // `ldd` (Linux), `otool -L` (macOS), or `dumpbin /DEPENDENTS` (Windows)
    // to show exactly which sub-dependency is causing the failure.
    
    print('Caught a dynamic library error!');
    
    // Prints the formatted cause, including CLI stdout/stderr from dependency checks
    print(e.toString());
  }
}
```

### 5. Advanced Usage & Bypassing Naming Conventions

Sometimes standard library naming (`lib*.so`) does not apply (e.g. versioned shared objects like `libfoo.so.1.0`), or you want to manually run an OS-level dependency inspection.

```dart
import 'dart:ffi';
import 'dart:io';
import 'package:dynamic_library/dynamic_library.dart';

void main() {
  // 1. Bypassing standard naming conventions
  // `loadDynamicLibrary` assumes 'lib' prefixes and standard extensions.
  // We bypass this entirely with `loadDynamicLibraryRaw` by supplying the explicit file name.
  try {
    DynamicLibrary rawLib = loadDynamicLibraryRaw(
      fileName: 'libonnxruntime.so.1.23.2',
      searchPath: '/usr/local/lib',
    );
    print('Loaded raw library successfully.');
  } on LoadDynamicLibraryException {
    print('Raw library not found. Expected in this isolated environment.');
  }

  // 2. Triggering OS Dependency Checks manually
  // You can run the cross-platform dependency resolution check yourself on any known file.
  String systemPath = Platform.isWindows 
      ? 'C:\\Windows\\System32\\kernel32.dll' 
      : '/bin/ls';
  
  if (File(systemPath).existsSync()) {
    // Manually invokes `dumpbin`, `otool`, or `ldd` depending on your OS
    ProcessResult result = callOSDependencyCheck(systemPath);
    
    if (result.exitCode == 0) {
      print('--- Dependencies for $systemPath ---');
      print(result.stdout);
    } else {
      print('Dependency check failed:\n${result.stderr}');
    }
  }
}
```
